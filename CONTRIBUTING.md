# Contributing to the documentation

## Writing rules

- Use **Wamule Development** for the business and **Wamule Operations Platform** for the software.
- Describe only verified current behavior. Label implementation requiring live confirmation and do not present planned portal/automation features as available.
- Use short, direct sentences and active verbs. State role boundaries and business consequences.
- Preserve financial, contract, audit, and data-correction accountability.
- Link to the authoritative feature or SOP rather than duplicating a rule.

## Review checklist

- [ ] Feature behavior verified against source and authorized test environment when necessary.
- [ ] No customer/production/secret data present.
- [ ] Media meets the Training Data policy.
- [ ] Sidebar, links, headings, and search checked.
- [ ] Business owner and technical verifier identified in the pull request.
- [ ] Release notes updated for material changes.

## Article conventions

Use one H1. Include an operational boundary when a feature is advisory, partial, privileged, or does not automatically change a related record. End each feature article with planned media types.

Use the templates in `templates/` for new SOPs, feature guides, and release notes.
